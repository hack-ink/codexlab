# Dep Roll

## Objective

Upgrade the repository's dependency graph to the newest set that remains mutually compatible with the toolchain, runtime, and verification gates. Manually edited requirements in non-generated manifests should stay at `X.Y` granularity so patch (`Z`) upgrades remain solver-controlled, lockfiles and generated dependency artifacts should be regenerated by tooling, and Dependabot should be treated as trailing reconciliation signal rather than the definition of success.

## When to use

- You are upgrading manifest-controlled dependency constraints or refreshing a stale dependency graph.
- You want one deliberate dependency roll to supersede or reconcile with existing Dependabot automation.
- Multiple ecosystems in one repository are involved (JavaScript/TypeScript, Python, Rust).
- The success condition is "latest verified compatible set", not "refresh lockfiles under the old constraints".

## Inputs

- Optional list of relevant manifests/lockfiles present in the repository.
- Installed tooling for any ecosystem being updated.
- Repo-native verification commands for each touched ecosystem.
- Existing Dependabot PR list or GitHub visibility for final reconciliation, if available.

## Scope discipline

- Keep the execution path proportional to the actual repo and request.
- If the repository or requested slice only involves one ecosystem, stay on that ecosystem instead of role-playing a cross-ecosystem rollout.
- Use the full multi-ecosystem sequence only when the repository or request actually spans multiple ecosystems.

## Version policy

- Target the latest versions that can pass together under the repo's actual compatibility constraints.
- Change manifest constraints first; lockfiles and other generated dependency artifacts follow from those manifests.
- For manually edited non-generated manifests, keep version intent at `X.Y` granularity and let `Z` float freely through resolver-driven upgrades.
- Canonical post-normalization forms for manually edited manifests are:
    - JavaScript/TypeScript (`package.json`): bare `X.Y` strings such as `"1.2"`, not `^1.2.3`, `~1.2.3`, or exact `1.2.3`, unless a documented exception requires tighter control.
    - Cargo (`Cargo.toml`): dependency entries should use explicit inline tables such as `{ version = "1.2" }` or `{ version = "1.2", features = ["derive"] }`. The `version` value itself should stay at bare `X.Y` granularity, not bare string shorthand like `"1.2"` for dependency entries, and not `^1.2.3`, `~1.2.3`, or exact patch strings such as `1.2.3` unless a documented exception requires tighter control. This style rule applies to dependency entries, not unrelated TOML keys.
    - Poetry (`pyproject.toml`): `~=X.Y` for `>=1.0`, and `X.Y.*` (or `==X.Y.*` when a PEP 508 string is required) for `0.x`.
- Avoid patch pins unless required; if required, document the reason and use explicit ranges like `>=X.Y.Z,<X.(Y+1)`.
- For `0.x` dependencies, prefer minor-capped ranges (`X.Y.*` semantics; use exact `==X.Y.*` when expressing a PEP 508 string).
- Never edit lockfiles or other generated dependency artifacts by hand; regenerate them with the ecosystem tool.

## Hard gates (non-negotiable)

- Do not hand-edit lockfiles or other generated dependency artifacts. Regenerate them with the ecosystem tooling.
- Do not confuse lock refresh with a manifest-constraint upgrade. `pnpm install`, `poetry update`, and `cargo update` are lock/regeneration steps, not manifest-upgrade steps.
- Do not proceed to Dependabot reconciliation until verification has run for the touched ecosystems.
- If the newest candidate set fails verification, back off to the newest set that passes and document the blocker instead of stacking unrelated fixes.
- If GitHub or Dependabot visibility is unavailable, report reconciliation separately as `blocked` or `skipped`; do not guess and do not let that silently redefine the upgrade outcome.
- If verification fails, stop and report the failure; do not "fix forward" by stacking unrelated changes.

## Procedure (repo-agnostic)

1. Inventory applicable ecosystems by checking for files:
    - JavaScript/TypeScript: `package.json` and `pnpm-lock.yaml` (if present)
    - Python: `pyproject.toml` and `poetry.lock` (if present)
    - Rust: `Cargo.toml` and `Cargo.lock` (if present)
      Only run commands and edit files for ecosystems that exist.

2. Inventory stale dependency signal before changing anything.
    - JavaScript/TypeScript:
        - Run `pnpm -r outdated` (or repo-equivalent filters) to see current vs latest for manifest-controlled workspace dependencies.
    - Poetry:
        - Run `poetry show --latest --outdated --top-level --format json`.
    - Cargo:
        - Run `cargo upgrade -n` to preview compatible manifest bumps for Cargo-managed dependencies.
        - If you need explicit signal on newer incompatible majors, optionally run `cargo upgrade -n --incompatible` and record them as deferred or blocked rather than silently ignoring them.

3. Upgrade manifest constraints first.
    - Normalize and bump constraints in each relevant manifest before treating lockfiles as settled.
    - Keep manually edited JS and Rust requirements in canonical `X.Y` form unless an exception is explicitly justified.
    - JavaScript/TypeScript:
        - Use a manifest-changing upgrade command such as `pnpm up -L -r` (or a targeted filtered variant) to move dependencies toward the latest desired versions.
        - After tool-driven bumps, normalize manually controlled `package.json` specifiers back to bare `X.Y` strings before final verification.
    - Poetry:
        - Update direct dependency constraints in `pyproject.toml` explicitly or via `poetry add`; do not treat `poetry update` as a manifest-upgrade command.
        - If manifests use Poetry version operators and the package is in `0.x`, normalize to `X.Y.*` (or `==X.Y.*` in PEP 508 strings) so patch upgrades remain unconstrained.
        - If manifests use Poetry version operators and the package is `>=1.0`, normalize to `~=X.Y` so patch upgrades remain unconstrained.
    - Cargo:
        - Use `cargo upgrade` to change `Cargo.toml` requirements.
        - After tool-driven bumps, normalize manually controlled Cargo requirements back to explicit inline-table dependency entries with `version = "X.Y"` before final verification, preserving any other dependency keys such as `features`, `default-features`, `workspace`, or `path` when they remain applicable.
        - If `cargo upgrade` is unavailable, bump `Cargo.toml` constraints manually and record that `cargo-edit` was missing; do not pretend `cargo update` performed the upgrade.
        - If the task is Rust-only, you can execute the Cargo slice end-to-end after inventory: preview with `cargo upgrade -n` (and optionally `cargo upgrade -n --incompatible`), bump direct `Cargo.toml` requirements with `cargo upgrade`, refresh `Cargo.lock` with plain `cargo update`, then run repo-native verification.

4. Regenerate lockfiles and dependency artifacts from the updated manifests.
    - Never resolve generated dependency files manually when a package manager can regenerate them from the manifest/source of truth.
    - JavaScript/TypeScript:
        - Ensure the workspace lockfile is synced after manifest changes. `pnpm up` may already rewrite the lockfile; if additional sync is needed, run `pnpm install`.
    - Poetry:
        - Re-resolve the lock from the updated constraints with `poetry update --lock`, adding repo-specific group flags only when that repository actually uses them.
    - Cargo:
        - Run plain `cargo update` after manifest changes to refresh the full `Cargo.lock`. This is the lockfile sync step, not the manifest-upgrade step.
        - Use `cargo update -w` only when you intentionally want a workspace-only lock refresh and the repository's scope makes that correct. Do not treat `-w` as the default full-lockfile refresh in a workspace.
    - If lockfiles require sync because an additional lockfile format is present, run:
        - `npm install --package-lock-only` (when `package-lock.json` exists).

5. Verify.
    - Run stack-relevant checks: linters, tests, and any repository-level validation scripts.
    - Re-run stale checks for the touched ecosystems and capture what is still blocked versus newly covered:
        - `pnpm -r outdated`
        - `poetry show --latest --outdated --top-level --format json`
        - `cargo upgrade -n`
    - If a package cannot remain at the newest available version while keeping the repo green, explicitly record the compatibility blocker, chosen fallback version, and proof that the fallback set verifies.
    - Confirm the resulting lockfile/solver output reflects the newest verified compatible full dependency set, not only the manually edited manifest lines.

6. Reconcile Dependabot PRs last.
    - Re-check open Dependabot PRs first by list (`gh pr list --state open --search "author:app/dependabot"`).
    - Treat reconciliation as reporting and cleanup, not the goal of the dep roll itself.
    - If GitHub or Dependabot visibility is unavailable, stop this step as `blocked` and report the missing access/tooling.
    - Classify each open PR as: covered by this change, blocked by constraints, or intentionally deferred.
    - If intentionally deferred, document the reason clearly in release/dependency notes.
    - If a dependency is intentionally capped but still noisy, either upgrade the capped line or add a documented `.github/dependabot.yml` ignore rule.

If your repo uses Cargo workspace dependencies:

- Preserve the repository's existing workspace dependency layout by default.
- Only change root/member dependency placement, `workspace = true`, or `path` usage when the repository already requires that style or the upgrade itself demands it.
- Do not rewrite Cargo workspace conventions solely because you are running `dep-roll`.

## Outputs

- List each manifest, lockfile, and generated dependency artifact changed (with file paths).
- Provide commands run per ecosystem and outcome (updated, no-op, blocked, or fallback-selected).
- Summarize the final dependency state as the newest verified compatible dependency graph, with explicit notes on anything still held back.
- Summarize remaining update signal with classification for each remaining PR (`covered`, `blocked`, or `intentionally deferred`).
- Call out any unresolved blockers (including version constraints, test/lint failures, or toolchain compatibility).

## Verification template

- Confirm manifest updates were applied before lockfile regeneration.
- Confirm manually edited non-generated version requirements stayed at `X.Y` granularity unless an exception was documented.
- Confirm Cargo dependency entries use explicit inline-table form with `version = "X.Y"` rather than bare string shorthand, and confirm those `version` values stayed at `X.Y` granularity.
- Confirm lockfiles and any related generated dependency artifacts were regenerated by tooling only.
- Confirm verification tasks ran for each touched ecosystem and include the exact commands.
- Confirm any fallback versions are justified by concrete compatibility evidence.
- Confirm Dependabot status, when available, aligns with the landed manifest state.

## Quick reference

- Detect ecosystems: `package.json`, `pyproject.toml`, `Cargo.toml`.
- Pre-checks:
  - `pnpm -r outdated`
  - `poetry show --latest --outdated --top-level --format json`
  - `cargo upgrade -n`
- Manifest-upgrade commands:
  - `pnpm up -L -r`
  - update `pyproject.toml` constraints explicitly or via `poetry add`
  - `cargo upgrade`
- Lockfiles and generated dependency artifacts (never hand-edit):
  - `pnpm install`
  - `poetry update --lock`
  - `cargo update`
- Reconcile Dependabot last: `gh pr list --state open --search "author:app/dependabot"`.

## Common mistakes

- Editing lockfiles or generated dependency artifacts by hand instead of regenerating.
- Treating `pnpm install` as if it upgraded direct dependency requirements.
- Treating `poetry update` as if it upgraded `pyproject.toml` constraints.
- Treating `cargo update` as if it upgraded `Cargo.toml` requirements.
- Using `cargo update -w` as the default full-lockfile refresh in a workspace and then misreading remaining Cargo PRs as uncovered work.
- Drifting manually edited manifest requirements away from `X.Y` intent and accidentally pinning patch releases.
- Leaving Cargo dependency entries in mixed styles after a dep roll, especially bare string shorthand alongside inline-table entries.
- Using inline-table Cargo dependency entries but drifting the `version` value itself away from bare `X.Y` granularity.
- Mixing ecosystems in one verification step (run verification for each touched stack).
- Treating a missing GitHub/Dependabot view as success instead of reporting reconciliation as `blocked`.
- Reconciling Dependabot before manifests/lockfiles are consistent.
- Stacking “fix forward” unrelated changes after a verify failure (stop and report instead).
